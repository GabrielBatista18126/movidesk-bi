Movidesk BI

Documentacao completa do sistema de Business Intelligence para Movidesk

O que e o sistema

O Movidesk BI e um sistema completo de Business Intelligence que extrai dados da API do Movidesk, transforma e armazena em um Data Warehouse PostgreSQL, e apresenta dashboards interativos via Streamlit.

ETL Automatico

Pipeline que roda a cada 5 minutos, extraindo dados da API do Movidesk, transformando e carregando no banco de dados.

100% Acuracia

Reconciliacao automatica garante que os dados do dashboard sao identicos aos do Movidesk. Entries deletadas sao removidas automaticamente.

Inteligencia

Previsao de estouro de contrato via projecao linear e score de risco (0-100) por cliente com classificacao automatica.

7 Dashboards

Visao Geral, Consumo de Contrato, Alertas, Produtividade, Tickets em Aberto, Inteligencia e Monitor ETL.

Pipeline ETL

Fluxo Completo

O pipeline e orquestrado pelo etl/main.py e segue esta ordem:

1. Persons (API) 2. Tickets + Actions (API) 3. Data Warehouse ├─ /persons?type=2 ├─ /tickets?$expand=... ├─ dim_tempo │ (clientes) │ clients, owner, actions, ├─ dim_clientes ├─ /persons?type=4 │ timeAppointments, createdBy ├─ dim_agentes │ (agentes) ├─ Extrai organizacoes ├─ fact_consumo └─ Upsert raw.* ├─ Extrai agentes └─ fact_tickets ├─ Transforma tickets ├─ Transforma time_entries 4. Machine Learning ├─ Upsert raw.* ├─ Previsoes consumo └─ Reconciliacao ├─ Score de risco (remove orfaos) └─ Alertas e-mail

Extractor (etl/extractor.py)

Responsavel por consumir a API REST do Movidesk com:

  • Paginacao automatica — busca em lotes de 50 registros (configuravel via PAGE_SIZE)
  • Retry com backoff — ate 3 tentativas com delay progressivo
  • Rate limiting — respeita header Retry-After em respostas 429
  • Modo incremental — usa watermark (lastUpdate gt {timestamp}) para buscar apenas tickets atualizados
  • OData v4 — datas no formato ISO direto (2026-04-08T08:05:31Z), sem wrapper datetime'...'

Endpoints consumidos

Endpoint$expandUso
/persons - Clientes (type=2) e Agentes (type=4)
/tickets clients($expand=organization), owner, actions($expand=timeAppointments,createdBy) Tickets, apontamentos de horas, organizacoes, agentes
Importante: O campo createdBy nos timeAppointments e sempre null. O agente correto vem de action.createdBy, por isso o $expand=createdBy e aplicado nas actions.

Transformer (etl/transformer.py)

Normaliza e limpa os dados brutos da API. Principais transformacoes:

TransformacaoDetalhe
Timezone (noon UTC) Datas de apontamento (timeAppointment.date) sao armazenadas como meio-dia UTC (12:00) para evitar shift de dia por timezone. A API retorna 2026-04-01T00:00:00 em horario local (BRT). Se armazenado como meia-noite UTC, em UTC-3 vira 31/03.
Organizacao Prioriza client.organization para agrupar por empresa. Fallback: se o proprio client e personType=2, usa ele como organizacao.
Agente Extrai de action.createdBy (quem criou a acao no ticket). Fallback: ticket.owner.
Horas Campo accountedTime (Double) — arredondado para 4 casas decimais. Ignora entries com hours <= 0.

Loader (etl/loader.py)

Persiste dados no PostgreSQL via upsert (INSERT ... ON CONFLICT DO UPDATE):

  • upsert_organizacoes() — tabela raw.organizacoes
  • upsert_clientes() — tabela raw.clientes (compatibilidade)
  • upsert_agentes() — tabela raw.agentes
  • upsert_tickets() — tabela raw.tickets
  • upsert_time_entries() — tabela raw.time_entries
  • reconcile_time_entries() — remove entries orfas (ver secao Reconciliacao)

Tambem gerencia o watermark (raw.etl_watermark) e o log de execucao (raw.etl_log).

Data Warehouse (etl/dw.py)

Popula a camada analitica em star schema:

dim_tempo

Calendario completo 2020-2030. Chave: YYYYMMDD. Campos: ano, semestre, trimestre, mes, semana, dia, dia_semana, e_fim_semana.

dim_clientes

Populada de raw.organizacoes + contrato vigente. Inclui plano, horas contratadas, vigencia.

dim_agentes

Populada de raw.agentes. Inclui nome, email, time, status.

fact_consumo

Uma linha por time_entry. JOIN com dims via organization_id. Campos: horas_gastas, mes_referencia.

fact_tickets

Uma linha por ticket. Campos: status, tipo, categoria, urgencia, dias_para_resolver, esta_aberto.

Machine Learning (etl/ml.py)

Previsao de Consumo

Projecao linear simples: taxa_diaria = horas_ate_agora / dias_passados, projetada para o total de dias do mes. Grava em analytics.previsoes_consumo.

Score de Risco (0-100)

Calcula um score composto para cada cliente com contrato:

ComponentePesoDescricao
Historico de estouros40%% de meses em que o cliente estourou o contrato
Tendencia30%Regressao linear sobre o % de consumo (crescimento)
Volatilidade20%Desvio padrao do consumo mensal
Urgencia tickets10%% de tickets High/Urgent no periodo

Classificacao:

0-24: BAIXO 25-49: MEDIO 50-74: ALTO 75-100: CRITICO

Reconciliacao de Dados

Garantia de 100% de acuracia. Quando um apontamento de horas e deletado no Movidesk, o ETL detecta e remove do banco automaticamente.

Dois modos de reconciliacao:

ModoQuando rodaO que faz
Full Load --full Compara TODOS os IDs de time_entries no banco com os da API. Remove qualquer ID que nao existe mais na API.
Incremental A cada ciclo (5 min) Para cada ticket atualizado, verifica se alguma entry foi deletada e remove do banco.

Scheduler (scripts/etl_scheduler.py)

Processo background que executa o ETL incremental automaticamente a cada N minutos.

# Padrao: a cada 5 minutos
python scripts/etl_scheduler.py

# Customizar intervalo
python scripts/etl_scheduler.py --interval 3

# Full load a cada ciclo
python scripts/etl_scheduler.py --full

# Rodar em background no Windows
start /B pythonw scripts/etl_scheduler.py

Dashboard

Paginas do Dashboard

O dashboard roda em Streamlit na porta 8501 com auto-refresh a cada 5 minutos e cache de queries com TTL de 2 minutos.

Visao Geral

Pagina principal com visao completa do mes atual:

  • Filtros — por data e por analista (funcionais, filtram todos os graficos e KPIs)
  • Banner de alerta — destaca maior cliente e analista com maior carga
  • KPIs — Total de horas, apontamentos, clientes ativos, analistas ativos
  • Horas por colaborador por dia — grafico de barras agrupadas (Plotly)
  • Horas por cliente — grafico de barras vertical
  • Tipo de problema — grafico donut (categoria + descricao)
  • Carga por analista — barras horizontais com % de anotacao
  • Prioridade — grafico donut (categoria + urgencia)
  • Detalhamento por cliente — tabela com horas, %, badge de risco (alto/medio/baixo), barra de progresso
  • Detalhamento por analista — tabela com horas, %, status, barra de progresso

Consumo de Contrato

  • Filtro por mes de referencia e por cliente
  • KPIs: total horas, clientes com lancamentos, tickets atendidos
  • Tabela de consumo por cliente (horas, tickets, lancamentos, ultimo lancamento)
  • Grafico de barras: horas consumidas por cliente (top 20)
  • Historico mensal de consumo (linha, top 10 clientes)

Alertas

  • Tabela de alerta de consumo com badges coloridos (ESTOURADO, CRITICO, ATENCAO, NORMAL)
  • Status automatico baseado no % de consumo vs contrato

Produtividade

  • KPIs do time: horas lancadas, tickets atendidos, agentes ativos, eficiencia media
  • Ranking de horas por agente (barras horizontais)
  • Eficiencia: tickets por hora (barras horizontais)
  • Tabela detalhada por agente (horas, tickets, clientes, tkts/hora, h/ticket)
  • Evolucao historica de horas por agente (linha)
  • Tempo medio por ticket historico (linha)

Tickets em Aberto

  • Lista de tickets em aberto ordenados por dias aberto
  • Top tickets com mais horas consumidas no mes

Inteligencia

  • Previsoes de consumo para o mes corrente
  • Score de risco por cliente (0-100) com classificacao
  • Sugestoes de upgrade de plano

Monitor ETL

  • Historico das ultimas 20 execucoes do ETL
  • Status (SUCCESS/FAILURE), duracao, registros processados

Operacao

Como Iniciar o Sistema

1. Configuracao (.env)

Crie um arquivo .env na raiz do projeto:

# Movidesk
MOVIDESK_TOKEN=seu_token_aqui
MOVIDESK_BASE_URL=https://api.movidesk.com/public/v1

# PostgreSQL
DB_HOST=localhost
DB_PORT=5432
DB_NAME=movidesk_bi
DB_USER=movidesk_user
DB_PASSWORD=sua_senha

# Alertas por e-mail (opcional)
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=seu@email.com
SMTP_PASS=senha_do_email
SMTP_FROM=seu@email.com
ALERT_EMAIL=gestor@empresa.com

2. Banco de dados

Execute os scripts SQL na ordem:

psql -d movidesk_bi -f sql/01_schemas.sql
psql -d movidesk_bi -f sql/02_raw_tables.sql
psql -d movidesk_bi -f sql/03_views.sql
psql -d movidesk_bi -f sql/04_contratos.sql
psql -d movidesk_bi -f sql/05_views_semana34.sql
psql -d movidesk_bi -f sql/06_star_schema.sql
psql -d movidesk_bi -f sql/07_inteligencia.sql
psql -d movidesk_bi -f sql/08_refactoring_v2.sql
psql -d movidesk_bi -f sql/09_filtros_consumo.sql
psql -d movidesk_bi -f sql/10_contratos_iniciais.sql

3. Primeiro full load

python -m etl.main --full

4. Iniciar scheduler + dashboard

# Terminal 1: Scheduler ETL (background)
python scripts/etl_scheduler.py --interval 5

# Terminal 2: Dashboard Streamlit
streamlit run dashboard/app.py --server.port 8501
O dashboard ficara disponivel em http://localhost:8501. O scheduler roda o ETL incremental a cada 5 minutos automaticamente.

Gerenciar Contratos

Os contratos ficam na tabela analytics.contratos. Para cadastrar um novo contrato:

INSERT INTO analytics.contratos (
    organization_id, client_id, plano_nome,
    horas_contratadas, vigencia_inicio, vigencia_fim,
    tickets_excluidos
) VALUES (
    '2034758315',           -- organization_id do Movidesk
    '2034758315',           -- client_id (mesmo valor)
    'Plano 40h',            -- nome do plano
    40,                     -- horas contratadas por mes
    '2025-01-01',           -- inicio da vigencia
    NULL,                   -- fim (NULL = indeterminado)
    ''                      -- tickets excluidos (separados por virgula)
);
Sem contrato cadastrado, o cliente aparecera como "SEM_CONTRATO" nos alertas. As horas continuam sendo contabilizadas normalmente — o contrato so afeta o calculo de % de consumo, alertas e previsoes.

Alertas por E-mail

O sistema envia 3 tipos de alerta automaticamente:

TipoQuandoDescricao
Estouro Consumo >= 80% do contrato Lista clientes que estouraram ou estao proximos de estourar
Previsao Projecao indica estouro Clientes que ainda nao estouraram mas a tendencia indica estouro ate fim do mes
Falha ETL Pipeline falha Detalhes do erro para investigacao

Configure as variaveis SMTP_* e ALERT_EMAIL no .env para ativar.

Troubleshooting

Dashboard mostrando dados antigos

  1. Clique no botao "Atualizar dados" na barra lateral (limpa o cache)
  2. Verifique se o scheduler esta rodando: tasklist | grep python
  3. Rode um ETL manualmente: python -m etl.main

Horas nao batem com o Movidesk

  1. Rode um full load com reconciliacao: python -m etl.main --full
  2. Isso remove entries deletadas no Movidesk que ainda estao no banco
  3. Verifique se nao ha entries orfas: compare totais por mes entre API e banco

ETL falhando com erro 400

  • Geralmente causado por formato OData incorreto. O sistema usa OData v4 (ISO direto)
  • Verifique o token no .env (MOVIDESK_TOKEN)

Agente aparecendo como "Sem agente"

  • Pode ocorrer em entries antigas importadas antes da correcao do $expand=createdBy
  • Solucao: python -m etl.main --full para reprocessar tudo
